
## OpenRouter API Key Issue Debugging (2026-03-24)

**Problem:** OpenRouter API key initially worked, but started returning 401 "User not found" errors via the Gateway. Direct API calls confirmed the key itself was valid.

**Investigation & Findings:**
- **Auth Profile:** `~/.openclaw/agents/main/agent/auth-profiles.json` was confirmed to have a valid `openrouter:default` entry.
- **Gateway Configuration:** `openclaw.json` fallbacks were updated to include OpenRouter, but Gateway did not seem to pick up the changes or use the correct profile.
- **Gateway Cache/Logic:** Suspected Gateway authentication cache issue or incorrect profile loading. `openclaw secrets reload` command was executed, but did not resolve the 401. Gateway logs did not show direct OpenRouter API calls being attempted by the Gateway itself.
- **Gemini Test:** `google/gemini-2.5-flash-lite` was successfully tested via `sessions_spawn` and confirmed to be working as a fallback.
- **Current Fallbacks:** `qwen-portal/coder-model` (working via OAuth) and `google/gemini-2.5-flash-lite` (working via API key) are currently functional. OpenRouter remains unusable via Gateway due to the ongoing 401 issue.

**Current Status:** OpenRouter API is not usable through the Gateway at this time due to persistent authentication errors. The root cause of the Gateway's inability to correctly load/use the OpenRouter credentials remains undetermined but is suspected to be related to caching or internal Gateway authentication handling.
